# CommitGate

이 저장소는 **CommitGate**를 쓴다. 코드 변경은 REQ 티켓 단위로 묶이고, Codex가 승인한 staged tree만 커밋된다.

**일반 구현으로 처리하지 마라.** 아래 계약을 먼저 읽어라.

## 계약 (정본)

저장소 루트의 **`AGENTS.md`**가 정본이다. 절대 규칙, 통제점 표, 승인 문장이 거기 있다.

> `AGENTS.md`에 `<!-- commitgate:contract -->` 마커가 없으면 그 파일은 CommitGate 계약이 아니다.
> 그 경우 `npx commitgate`가 저장소 루트에 **`AGENTS.commitgate.md`**(계약 템플릿 사본)를 함께 설치해 둔다.
> 그 파일을 계약으로 읽고, 사용자에게 `AGENTS.md`로의 병합을 요청하라.

## 요구사항 받기

사용자 요청이 아래 네 칸으로 정리되지 않으면 먼저 물어라. 추측하지 마라.

```text
- 무엇을:
- 왜:
- 제약:
- 완료 기준:
```

정리되면 `req:new`로 티켓을 만든다.

## 진행 방법 — `req:next`가 시키는 대로

**다음 행동을 스스로 추측하지 마라.** 도구가 상태에서 계산해 준다.

```sh
req:next <REQ-id>
```

> 워크플로 명령은 이 저장소의 **패키지매니저 실행 형식**으로 돌린다(npm은 `run`과 `--` 구분자가 필요하고, pnpm·yarn은 스크립트 이름을 바로 받는다).
> `req:next`의 `RUN` 출력이 언제나 정확한 형태를 그대로 보여 주므로, 그 줄을 복사해 쓰면 된다.

출력의 `kind`가 정본이다(`--json`으로 기계 판독 가능).

| kind | 할 일 |
|---|---|
| `RUN` | 출력된 명령을 **그대로** 실행하고, 다시 `req:next` |
| `AGENT` | 도구가 대신 못 하는 작업(구현·문서 작성·`git add`). 하고 나서 다시 `req:next` |
| `AWAIT_HUMAN` | **멈춘다.** 출력된 승인 문장을 사용자에게서 **그 문장 그대로** 받기 전에는 진행하지 않는다 |
| `DONE` | 이 티켓에서 도구가 할 일이 없다. 통합은 별도 통제점 |
| `BLOCKED` | 사람에게 보고한다. 같은 리뷰를 재시도하지 마라 |

이 루프를 **끊지 말고** 반복한다. `AWAIT_HUMAN`·`BLOCKED`·오류에서만 멈춘다.

## 반드시 지킬 것

- 리뷰 대상은 `git add` 한 파일뿐이다.
- `state.json`과 `responses/`는 직접 `git add` 하지 않는다 — 도구가 관리한다.
- `req:review-codex`가 exit 3(NEEDS_FIX)이면 findings를 수정하고 재리뷰한다.
- exit 2(BLOCKED)면 **같은 리뷰를 재시도하지 말고** 사람에게 보고한다.
- 승인은 받은 문장 그대로만 유효하며, 다음 통제점으로 **이월되지 않는다**.
