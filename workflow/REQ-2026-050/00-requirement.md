# REQ-2026-050 요구사항

design 리뷰 phase 분해 점검 기준 + `sync --persona` drift 선택 경로

## 배경 — 소비자 프로젝트에서 관측된 실패

소비자 저장소(`44_yammy_sales`, commitgate@0.9.8) 운영 실측에서 다음이 관측됐다. 해당 저장소는 **읽기 전용 분석 fixture**이며 이 REQ는 그 저장소를 수정하지 않는다.

- design 리뷰가 수렴하지 않아 티켓 3건이 연속 폐기됐다(REQ-001→002→003, `successor_of` 체인). 리뷰 호출 20회·48.2분·wall-clock 2시간 10분을 쓰고 산출물 0.
- 폐기 사유는 티켓 자신의 `human_resolution` 기록이 명시한다: *"한 티켓이 네 개의 독립 계약을 동시에 지고 있어 설계 리뷰가 수렴하지 않는다."*
- 설계 승인 후 phase 경계가 무너져 재분해된 사례가 완주 티켓 5건 중 4건에서 발생했다(design series #2, 추가 8라운드). 그 과정에서 phase 리뷰 4라운드가 폐기됐다.

## 문제 정의

`02-plan.md`는 이미 design 리뷰 프롬프트의 **권위 아티팩트로 전문 포함**된다(`assembleReviewPrompt`). 실제로 리뷰어는 계획을 읽고 지적한다(`file: 02-plan.md` findings 다수).

문제는 계획이 리뷰에 **안 들어가는 것**이 아니라, **phase 분해 품질을 심사할 기준이 없는 것**이다. 현재 `workflow/review-persona.md`의 `REVIEW_KIND: design` 점검 목록 6개(요구사항·문서 간 모순·계약 위반·oracle·보안·호환성) 어디에도 phase 책임 경계·의존성·독립 검증 가능성 항목이 없다.

## 요구사항

1. 패키지 기본 `workflow/review-persona.md`의 design 점검 관점에 phase 분해 심사 기준을 추가한다.
2. 신규 `commitgate init` 프로젝트에는 이 정책이 자동 적용된다.
3. 기존 프로젝트의 persona 파일을 **무단으로 덮어쓰지 않는다**.
4. 기존 프로젝트를 위한 **명시적 opt-in 갱신 경로**를 제공한다. 사용자 편집이 있을 수 있는 파일은 diff를 보여주고 사용자 선택을 요구한다.

## 비목표

- 문서 줄 수·`prompt_bytes`·계약 수를 하드 차단 기준으로 삼지 않는다. 실측상 초기 프롬프트 크기는 폐기 3건 중 2건만 분리했고(폐기된 REQ-001 37.1KiB < 수렴한 REQ-006 37.7KiB), 크기는 약한 선행지표일 뿐 규칙이 될 수 없다.
- P1 정의(카테고리·정상 경로·증거 3요소)를 넓히지 않는다. 분해 관점 추가가 차단 기준 확대로 읽히면 승인이 나지 않게 되고, 그것은 리뷰의 실패다.
- 내구성 원장·리뷰 호출 lifecycle·예외 사유 구조화·lockfile 프롬프트 축소는 이 REQ의 범위가 아니다(각각 별도 REQ).

## 인수 기준

- 기본 persona에 phase 분해 점검 기준이 존재하고, 그 존재를 테스트가 검증한다.
- `sync --persona`가 기존 동작(파괴적 쓰기 0건)을 기본값으로 유지한다.
- 명시적 opt-in 플래그 없이는 어떤 경로로도 기존 persona가 교체되지 않는다.
- opt-in 교체 시 백업이 남는다.
- `44_yammy_sales`에 파일 변경이 0건이다.
